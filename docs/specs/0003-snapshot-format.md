# 0003. Snapshot Format

## Scope

Phase 2 introduces snapshot metadata persistence. It records the state of a source tree at a point in time by reusing the Phase 1 block ingest result.

Included in Phase 2:

- `snapshots/` repository directory
- `indexes/snapshot-index.json`
- snapshot JSON files
- snapshot create/list/detail commands
- minimal snapshot UI

Excluded from Phase 2:

- snapshot comparison
- restore
- snapshot deletion
- block garbage collection
- integrity verification UI
- SQLite migration

## Repository Layout Additions

Phase 2 extends the repository layout as follows:

```text
chrona-repository/
  manifest.json
  blocks/
  snapshots/
    20260619T103000Z_8f31c2.json
  indexes/
    snapshot-index.json
  logs/
```

Rules:

- New repositories should create `snapshots/` during repository creation.
- Opening an existing Phase 1 repository should create `snapshots/` and `indexes/snapshot-index.json` if missing.
- Missing snapshot index is treated as an empty index, not repository corruption.

## Snapshot ID

Snapshot IDs use this format:

```text
YYYYMMDDTHHMMSSZ_<6-char-uuid-prefix>
```

Example:

```text
20260619T103000Z_8f31c2
```

Rules:

- Snapshot IDs are generated by the application.
- Snapshot IDs accepted from UI/API must contain only ASCII letters, digits, `_`, and `-`.
- Snapshot file path is always `snapshots/{snapshotId}.json`.
- Snapshot ID path traversal is rejected.

## Snapshot JSON

```json
{
  "schemaVersion": 1,
  "id": "20260619T103000Z_8f31c2",
  "name": "Initial import",
  "createdAt": "2026-06-19T10:30:00Z",
  "sourceRoot": "/Users/example/Documents/demo",
  "summary": {
    "fileCount": 12,
    "totalOriginalBytes": 7340032,
    "totalBlockReferences": 21,
    "newBlockCount": 8,
    "reusedBlockCount": 13,
    "newStoredBytes": 4194304
  },
  "files": [
    {
      "relativePath": "notes/a.txt",
      "sizeBytes": 1024,
      "modifiedAt": "2026-06-19T10:00:00Z",
      "blocks": [
        {
          "index": 0,
          "offset": 0,
          "sizeBytes": 1024,
          "hash": "sha256hex",
          "wasNew": true
        }
      ]
    }
  ]
}
```

## Snapshot Index JSON

```json
{
  "schemaVersion": 1,
  "snapshots": [
    {
      "id": "20260619T103000Z_8f31c2",
      "name": "Initial import",
      "createdAt": "2026-06-19T10:30:00Z",
      "sourceRoot": "/Users/example/Documents/demo",
      "fileCount": 12,
      "totalOriginalBytes": 7340032,
      "newStoredBytes": 4194304
    }
  ]
}
```

Rules:

- List commands read from `snapshot-index.json`.
- Detail commands read the full snapshot JSON.
- Snapshot index is sorted newest first by `createdAt`.
- Snapshot index writes use `.tmp` then rename.
- Snapshot file writes use `.tmp` then rename.

## Phase 1 Model Change Required

`FileIngestResult` must gain this field:

```rust
pub modified_at: String
```

TypeScript must mirror it:

```ts
modifiedAt: string;
```

Reason: snapshot files need file modification time, and Phase 1 `ScannedFile` already captures it.

## Commands

Phase 2 commands:

```rust
create_snapshot(repository_path: String, source_path: String, name: String) -> Snapshot
list_snapshots(repository_path: String) -> Vec<SnapshotIndexItem>
get_snapshot(repository_path: String, snapshot_id: String) -> Snapshot
```

Command behavior:

- `create_snapshot` runs Phase 1 block ingest, then persists snapshot metadata.
- `list_snapshots` returns index items only.
- `get_snapshot` returns the full snapshot file.
- Commands return user-facing error strings through Tauri, following Phase 1 command style.

## UI Requirements

Phase 2 UI should add:

- snapshot name input
- `Create Snapshot` action
- snapshot list panel
- snapshot detail panel

The Phase 1 direct block ingest result can remain visible for development, but the primary workflow should become snapshot creation.
